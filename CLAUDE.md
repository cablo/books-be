## Project overview

A Micronaut (4.4.2) + Kotlin REST API ("Books Application") that manages `Person` records and their associated `Book` records, backed by PostgreSQL. Built with Gradle Kotlin DSL, JDK 21.

## Commands

All commands use the Gradle wrapper (`gradlew.bat` on Windows / `./gradlew` in bash).

- Run the app: `./gradlew run` (starts on port 8080)
- Build: `./gradlew build`
- Run all tests: `./gradlew test`
- Run a single test: `./gradlew test --tests "cz.cablo.books.SomeTestClass"`
- Regenerate `openapi.yml`: happens automatically during compilation (KSP annotation processor writes it to the project root — see `micronaut.openapi.target.file` in `build.gradle.kts`); don't hand-edit it.

There are currently no test sources in `src/test` — if you add tests, they go under `src/test/kotlin/example/micronaut/...` mirroring the main package layout.

### Local Postgres database

The app requires PostgreSQL to run (Flyway migrations run against it on startup). Use the provided scripts to run the official `postgres` image in Docker (see [POSTGRES-DOCKER.md](POSTGRES-DOCKER.md)):

- Start: `powershell -ExecutionPolicy Bypass -File .\scripts\run-postgres.ps1`
- Stop: `powershell -ExecutionPolicy Bypass -File .\scripts\stop-postgres.ps1`

Connection details (also the defaults baked into `application.yml`): `jdbc:postgresql://localhost:5432/app_db`, user `app_user` / password `app_password`. Override via `DATASOURCE_URL`, `DATASOURCE_USERNAME`, `DATASOURCE_PASSWORD` env vars.

## Architecture

Standard layered Micronaut structure under `src/main/kotlin/example/micronaut/`:

- `controller/` — HTTP layer (`@Controller`). `PersonController` exposes `/persons` (GET list, POST create) and only talks to the service layer, never repositories directly.
- `service/` — `@Singleton` business logic with `@Transactional` boundaries (`PersonService`). Owns the mapping between JPA entities and API DTOs (private `toResponse()` extension functions live here, not in the controller or DTOs).
- `repository/` — `io.micronaut.data.jpa.repository.JpaRepository` interfaces (compile-time implemented by micronaut-data via KSP). No custom query methods yet.
- `domain/` — JPA entities (`Person`, `Book`). `Person.books` is the owning-side-free end of a bidirectional `@OneToMany`/`@ManyToOne` (`Book.author`); always add books through `Person.addBook()` so both sides of the association stay in sync — don't push to `person.books` directly.
- `api/` — request/response DTOs (`@Serdeable` data classes) plus `OpenApiConfiguration` (the `@OpenAPIDefinition` used to generate `openapi.yml`).

### Persistence

- Schema is managed by Flyway migrations in `src/main/resources/db/migration/` (`V1__init_schema_and_seed_data.sql` creates `persons`/`books` and seeds 10 persons with 5 books each). Hibernate `hbm2ddl.auto` is `none` — all schema changes must go through a new Flyway migration (`V2__...sql`, etc.), never by editing `V1` or relying on Hibernate DDL.
- Dialect is PostgreSQL-specific (`VARCHAR`, `BIGINT GENERATED BY DEFAULT AS IDENTITY`) — keep new migrations Postgres-compatible.
- `books.isbn` has a unique constraint; `books.person_id` is a required FK to `persons.id`.

### API surface

`openapi.yml` at the repo root is the source of truth for the HTTP contract (generated from the annotated controller/DTOs). Currently: `GET /persons` (list all persons with nested books) and `POST /persons` (create a person with an optional list of books in one request).
